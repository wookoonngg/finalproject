# Producer–Consumer Problem

## 1. 개요

첫 번째 실습에서는 생산자-소비자 동기화 문제를 해결하기 위하여 아래의 세 가지 방법을 구현하고 비교하였다.

1. 조건변수를 활용한 방식
2. 세마포어를 활용한 방식
3. 세마포어를 사용하였을 때 데드락 상황

이를 통해 조건변수와 세마포어를 사용하였을 때의 동작 원리와 차이점, 그리고 데드락 발생 조건을 확인하는 것을 목표로 한다.

## 2. 조건변수 기반 구현

### 2.1 동작 방식(원리)

조건변수 방식에서는 아래와 같은 요소를 사용하였다.

- `mutex` : 임계구역 보호
- `empty_cv` : 버퍼가 다 차 있을 때 생산자 대기
- `fill_cv` : 버퍼가 비었을 때 소비자 대기

조건변수에서 사용한 핵심 특징을 아래와 같이 정리한다.

- **while**: 조건을 `if`가 아닌 `while`문을 활용하여 찰나의 순간 조건이 변하여 나타나는 에러 방지
- **wait()**: mutex를 자동으로 반납하고 sleep 상태로 진입
- **notify()**: 대기 중인 스레드를 깨우는데, 위의 두 가지 변수 선언으로 명확하게 생산자는 소비자를, 소비자는 생산자를 깨움

### 2.2 실행 결과

- 버퍼 크기가 0 ~ 30 범위 내에서 유지됨
- 생산자와 소비자가 번갈아 가며 실행됨
- 버퍼의 overflow, underflow 발생하지 않음

### 2.3 평가

**장점**
- 조건을 확인한 후에 가시적으로 sleep/wakeup 제어 가능 → 프로그래머에게 친화적인 방식같다고 느껴졌음
- 큐에서 자기 때문에 busy waiting 없음
- 상태에 따라 동기화되기에 보다 직관적

**단점**
- 조건변수 2개를 사용하기에 구현 시에 복잡함을 느낌
- 잘못 구현하게 된다면 signal 누락 가능

## 3. 세마포어 기반 구현

### 3.1 동작 원리

임계구역 보호를 위한 mutex와 빈 공간 개수와 채워진 데이터 개수를 나타낼 수 있는 변수 두 가지를 사용하였다.

- `mutex`: 임계구역 보호 (binary semaphore)
- `empty`: 빈 공간 개수
- `full`: 채워진 데이터 개수

위의 세 가지 인스턴스를 활용하여 핵심 흐름은 아래와 같다.

```
생산자:
empty.acquire()
mutex.acquire()
생산
mutex.release()
full.release()

소비자:
full.acquire()
mutex.acquire()
소비
mutex.release()
empty.release()
```

### 3.2 실행 결과 분석

- empty/full을 통해 카운팅만 진행하여 상태를 관리하였음
- 조건 체크 하지 않고 카운팅 변수만으로 점유 대기

### 3.3 평가

**장점**
- semaphore 하나만 관리하면 되었기에 구현이 간결
- 카운팅 기반이라 보다 빨리 진행된다고 생각함

**단점**
- 단순히 카운팅으로 제어를 하기 때문에 만약 논리 흐름을 놓치면 오류가 발생함
- 실제로 코드 구현 시에 조건변수에 비하여 더 많은 시간을 투자함
- 디버깅도 어려웠음
- 카운팅에 의존하여 프로세스가 자원을 점유한 상태로 만약 함수 실행 로직이 어긋나면 조건변수에 비하여 데드락 발생 가능성이 높을 것으로 판단하였음

이에 대하여 데드락이 발생하는 경우를 세마포어 변수를 활용하여 (생산자-소비자 문제와는 별개) 구현해 보았다.

## 4. Deadlock 발생 가능 실습

### 4.1 코드 동작

프로세스는 두 개(P, Q)인 상황에서

```
Process P:
  A 획득 → B 요청

Process Q:
  B 획득 → A 요청
```

일 때 실행 흐름은 P가 먼저 A를 점유하고 Q는 B를 점유한다. 이후 P는 B를 요청하지만 무한히 대기하게 되고 Q는 A를 요청하지만 무한히 기다리게 된다. 즉 서로가 서로를 무한히 기다리는 Deadlock이 발생한다.

### 4.3 데드락 발생 조건

본 코드에서는 아래의 데드락 발생 조건이 모두 만족됨을 알 수 있다.

1. **상호 배제 (Mutual Exclusion)**: 자원 A, B는 동시에 하나의 프로세스만 사용 가능
2. **점유 대기**: P는 A를 잡고 B를 기다리며, Q는 B를 잡고 A를 기다림
3. **비선점**: 자원을 강제로 빼앗을 수 없음
4. **순환대기**: P – Q – P 형태로 사이클 발생

## 5. 조건변수 vs 세마포어 비교

개발하는 입장에서 걸린 시간 및 복잡성, 난이도 차이는 있었으나 이를 차치하고 비교할 수 있는 항목을 도식화하여 비교하고자 한다.

| 항목 | 조건 변수 | 세마포어 |
|---|---|---|
| 개념 | 상태 기반 | 자원 개수 기반 |
| 구현 난이도 | 조건 작성 후엔 로직 구성이 다소 수월했음 (개인적) | 호출이 자주 꼬임 (개인적) |
| 직관성 | 높음 | 낮음 |
| 성능/효율 (시간적) | 안정성 | 효율 |
| 데드락 위험 | 비교적 낮다고 판단 | 높음 |
| 사용 | 조건이 중요한 문제 혹은 순서가 무조건적으로 보장되어야 할 때 | 단순 자원 제한이나 빠른 결과가 필요할 때 |

세마포어의 경우 순서 보장을 완벽히 하지 않는다. 데드락 코드만 보아도 프로세서가 자원을 할당받고자 하는데 이에 대한 순서가 명확히 되지 않아 데드락이 발생한다. 또한 점유 상태를 계속 유지한다는 특징이 있다. acquire 하게 되면 이미 잡은 자원을 놓지 않고 다음 자원을 기다린다. 즉 세마포어는 전체 시스템 상태나 논리와는 무관하게 그 당시의 카운트를 보고 자원중심적으로 동작한다고 느꼈다.

이에 반하여 조건변수의 경우 조건이 확실히 들어맞을 때만 실행되기에 보다 논리적으로 동기화 문제를 해결하는 방식이라 판단하였다. 조건변수의 경우 wait를 호출하면 자연스럽게 mutex를 내려놓게 된다. 즉 상태를 스스로 인식할 수 있고 이로 인해 자원 접근에 대한 순서 제어가 가능했다.

그렇다면 어떤 방식이 더 효과적인가? 라고 물었을 때 단편적으로 하나를 꼽긴 어렵다고 판단한다. 대신 상황별로 보다 특화적인 방식이 존재한다고 생각한다. 조건이 중요한, 예를 들어 위와 같은 생산자-소비자 문제, 버퍼 상태를 체크해야 하는 문제, 특정 이벤트를 기다리는 문제에 있어선 조건변수가 보다 좋은 성능을 낼 것이라 생각한다. 그 이유는 상태 기반으로 안전하게 실행된다고 판단했기 때문이다. 찾아본 결과 실제로 조건 동기화는 대부분의 고급 동기화에서 사용된다고 한다. 세마포어의 경우 단순하게 자원을 제한하는 경우에 보다 잘 쓰일 수 있다고 생각한다. 스레드 개수를 제한하거나, 동시 접근 수를 제한할 때 빠르고 간단하게 카운팅에 쓰면 좋을 것으로 판단하였다.

## 6. 결론 및 느낀점

생산자 소비자 실습을 통해 3가지의 문제 중 가장 많은 시간을 썼던 동기화 개념이었으나 가장 기본적인 로직을 잡을 수 있어 동기화에 대한 이해도를 확실히 높일 수 있었다.

특히 조건변수 방식은 while문에 조건을 명확히 작성해보면서 "조건이 만족될 때까지 기다린다"라는 개념을 명확하게 실습하였고, 세마포어 방식은 자원 개수를 기반으로 동작하는 효율성을 노린 방식임을 확인할 수 있었다.

또한 세마포어를 실습할 동안 계속해서 로직이 꼬인 경험이 있는데, 이러한 경험을 통해 세마포어를 잘못 사용할 경우 매우 쉽게 데드락 등의 문제가 발생할 수 있음을 체감하였다. 강의시간에 들은 바로는 ignore 방식을 쓸 만큼 데드락 문제는 빈번히 발생하지 않는다고 배웠는데, 내 코드창에선 단순히 자원 획득 순서만 바꾸어도 시스템이 멈춰버리는 상황이 발생하여 꽤 당황스러웠다.

마지막으로 이 실습을 통해선 이러한 교훈(나름의 생각?)을 얻었다.

- 동기화는 단순 구현보단 설계가 중요할 수 있겠다.
- 세마포어를 사용하는 것보단 차라리 자원획득 순서를 통일해버리는 게 데드락을 예방하는 데 효과적일 수 있겠다.

결론적으로 교수님께서 상시 말씀해 주신 바와 같이 모든 방식은 이분법적으로 좋고 싫고가 있는 것은 아닌 것 같다. 상황에 따라 적절한 동기화 방식을 선택하는 것이 중요할 것이며, 구현은 못다했지만 세마포어를 조건변수로서 사용하게 되면 보다 더 좋은 성능을 낼 수 있을 것 같다는 생각이 들었다.

# Readers–Writers Problem

## 1. 개요

세마포어 변수를 활용하여 다중 스레드 환경에서 발생할 수 있는 독자-저자 문제를 구현한다. 핵심 과제로는 데이터가 변하지 않도록 일관성을 유지하면서 여러 독자가 동시에 자원에 접근할 수 있도록 허용하는 독자 우선 방식을 구현하는 것과, 작성자가 대기 중일 때 독자의 진입을 차단하여 작성자의 기아 현상을 방지하는 작성자 우선 방식의 구현이다.

## 2. 독자 우선 방식

### 2.1 동작 원리

독자 우선 방식에서는 여러 독자가 동시에 데이터를 읽을 수 있도록 허용하고, 작성자는 독자가 모두 나갈 때까지 대기해야 한다.

주요 변수는 아래와 같다.

- `read_count`: 현재 읽고 있는 독자 수
- `rmutex`: read_count 가드
- `resource_mutex`: 공유 자원 그 자체 가드

핵심 흐름은 아래와 같다.

- 독자들은 read_count에 대한 락을 잡고 카운트를 올린다
- 첫 번째 독자가 자원에 lock을 획득한다
- 이후 독자들은 동시에 모두 접근이 가능하다
- 마지막 독자는 자원에 대한 lock을 해제하게 되고 작성자가 들어올 수 있게 된다

### 2.2 실행 결과

- 여러 독자가 동시에 읽는 모습이 확인 가능하다
- 작성자는 독자가 모두 읽은 후에 들어온다

### 2.3 문제점

당연 독자 우선 방식에선 코드 실행 결과에서도 볼 수 있듯 독자가 모두 읽은 후에나 작성자가 입장 가능하기에 작성자에 대한 starvation이 발생한다. 간단한 구현을 위해 작성자, 독자를 모두 3으로 맞추었지만 독자를 100명으로 설정하고 진행한 테스트에선 기아 현상을 보다 확실하게 느낄 수 있었다.

## 3. 작성자 우선 방식

### 3.1 동작 원리

작성자 우선 방식에서는 작성자가 대기 중일 경우 새로운 독자의 진입을 차단한다.

구현에 사용된 주요 변수는 아래와 같다.

- `write_count`: 현재 대기 중인 작성자 수
- `wmutex`: write_count 가드
- `read_try`: 독자 막는 가드
- `resource_mutex`: 공유 자원 보호

구현 핵심 흐름은 아래와 같다.

- 첫 번째 작성자가 read_try lock을 잠가 독자를 차단한다
- 작성자가 끝날 때까지 독자 진입을 금지한다
- 모든 작성자가 끝나면 독자를 다시 허용한다

### 3.2 실행 결과 분석

- 작성자가 연속적으로 실행된다
- 독자는 작성자 작업 이후에 수행된다
- 데이터 일관성 유지에 유리하다고 보인다

### 3.3 문제점 (starvation)

작성자 우선 방식에서는 독자 기아 현상이 발생한다. 아래는 작성자 스레드 개수를 100개로 늘렸을 때의 경우이다.

## 4. 비교 및 평가

| 항목 | 독자 우선 | 작자 우선 |
|---|---|---|
| 동시 읽기 | 가능 | 제한 |
| 쓰기 처리 | 느림 | 빠름 |
| 기아 발생 | 작성자 기아 | 독자 기아 |
| 구현 복잡도 | 낮음 | 높음 |
| 사용 | 읽기 중심 시스템 | 데이터 일관성이 중요한 경우 |

코드를 작성하며 느낀 바로는 두 가지 방식 다 starvation이 발생하기 때문에 무엇이 더 성능적으로 좋고 나쁘다를 판단하는 것은 무의미하다고 생각이 든다. 그저 상황에 따라 보다 우위에 있는 방식을 선택하는 것이 중요하다고 생각한다. 독자 우선 코드의 경우 독자에 대하여 db 자체에 대한 접근만 제어하면 되기에 구현이 비교적 간단하고, 보통의 어플리케이션이나 웹의 경우 사용자는 독자의 위치인 경우가 많기에 읽기 위주의 시스템에서 유용히 쓰일 수 있다고 판단하였다. 반대로 작성자 우선 방식의 경우 독자에 대한 가드도 필요했기에 보다 구조는 복잡하다고 느꼈다. 또한 독자 진입 자체를 차단하기에 작성자가 수정 시에 데이터가 보다 안전하게 보호될 수 있다고 판단하였다. 그렇기에 데이터의 일관성이 중요한 경우에 작자 우선 방식이 사용될 것이라 판단하였다.

## 5. 결론 및 느낀점

이번 실습은 사실 이론만 배웠을 땐 가장 쉽게 학습한 항목이고 동기화 문제를 공부할 때 본인의 스트레스를 덜어준 항목이었기에 과제 공개된 이후에 가장 먼저 실습에 임한 파트였다. 하지만 슈도 코드로 대충 로직을 작성할 땐 분명 쉬운 논리였으나 막상 락을 잡으려 하니 처음엔 작성자가 아예 나타나지도 않는 시행착오를 겪기도 하였고, 공부가 덜 된 상태였는지, 작성자 우선 코드에서 reader count를 작성하여 괜히 mutex를 더 추가하여 스스로 난이도를 올리기도 하였다.

결론적으로 이 파트를 실습하며 느낀 바는 독자 우선 방식은 읽기 성능이 뛰어나지만 작성자 기아 상태에 빠질 수 있으며, 작성자 우선 방식은 데이터 일관성을 보장하지만 독자의 응답성이 저하될 수 있다. 가장 인상적인 것은 두 방식 모두 완벽하지 않다는 점이다. 어느 한쪽을 우선하면 다른 한쪽에서 기아 문제가 발생하기 때문에 실제 시스템에서는 두 방식을 절충한 공정성 기반 알고리즘이 필요하다고 생각이 든다.

그래서 이러한 공평한 알고리즘이 현재 없는가? 궁금하여 찾아본 결과 FIFO 뮤텍스 방식이 있다고 한다. 임계구역 직전에 줄서기 위한 락을 하나 더 두고, 작성자가 대기 중이더라도 먼저 들어와서(aging 기법 활용) 오래된 독자들이 있으면 그 독자들까지 접근을 허용하고, 그 뒤에 온 독자들은 작성자 뒤에 들어가서 줄을 서게 한다.

또한 현대 운영체제에서는 내부적으로 기아 문제를 방지하는 대기 큐가 내장된 rwlock을 제공한다고 한다. 이를 독자와 작자가 번갈아 가며 들어올 때 핑퐁 스타일로 쌓이게 만들어 독자나 작자가 아무리 밀려들어와도 큐 중간에 다른 입장인 스레드가 끼어있으면 그 뒤의 시스템은 양보하게끔 강제 조정을 한다고 한다. 기회가 되면 이 라이브러리에 대하여도 구현을 해보고 싶다는 생각이 들었다.

# Dining Philosophers Problem

## 1. 개요

본 실습에서는 식사하는 철학자 문제를 해결하기 위한 다음의 세 가지 방법을 구현한다.

1. 한 번에 자원을 요구하기 (점유 대기 차단)
2. 자원 순서 할당 방식
3. Banker's Algorithm 기반 회피 방식

각 방법을 통해 데드락 방지 기법을 직접 구현하고 실행 결과를 비교 분석하도록 한다.

## 2. 한 번에 요구하기 방식 (All-at-once)

### 2.1 동작 원리

이 방식은 철학자가 식사를 시작하기 전에 두 개의 포크를 한 번에 확보하도록 한다.

주요 특징은 아래와 같다.

- `room_once` 세마포어를 통해 포크 획득(2개)을 원자적으로 수행
- 두 포크를 모두 얻기 전까지 다른 철학자의 접근 제한
- 점유 대기 조건 제거

### 2.2 실행 결과 분석

- 동시에 여러 철학자가 식사하지 못함 (직렬화가 됨)
- 데드락 발생하지 않음
- 자원 활용 효율이 낮을 것 같음

### 2.3 평가

**장점**
- 구현이 가장 간편하였음
- 데드락을 온전히 방지할 수 있음

**단점**
- 직렬적으로 수행하고 있다고 판단되기에 병렬성이 감소하여 성능이 저하됨
- 불필요하게 더 많은 대기가 발생함

## 3. 자원 순서 할당 방식

### 3.1 동작 원리

모든 철학자가 동일한 순서로 자원을 획득하도록 하여 순환대기를 차단한다.

구현 방식은 아래와 같다.

- 대부분 철학자는 왼쪽에서 오른쪽
- 마지막 철학자는 오른쪽에서 왼쪽으로

자원 획득 순서를 깨뜨려 순환구조를 제거한다.

### 3.2 실행 결과 분석

- 여러 철학자 동시에 식사 가능
- 데드락 발생하지 않음
- 병렬 처리 가능

### 3.3 평가

**장점**
- 병렬성 유지 가능
- 성능 비교적 우수

**단점**
- 설계가 처음에 ordering에서 mod 연산 구현에 애를 먹음
- 특정 철학자 기아 발생 가능

## 4. Banker's Algorithm

### 4.1 동작 원리

은행원 알고리즘은 자원을 할당하기 전에 미리 시스템이 안전한 상태인지 예측하고 확인하는 방식이다.

본 코드에서 구현은 아래와 같다.

- 전체 포크 수 관리 (`available_forks`)
- 철학자가 2개의 포크를 요청할 수 있을 때만 허용
- 부족할 경우 대기

### 4.2 실행 결과 분석

- 항상 안전 상태 유지됨
- 중앙에서 제어하는 방식

### 4.3 평가

**장점**
- 데드락 완전 회피 가능
- 안정성이 높음

**단점**
- 구현이 복잡
- 모든 요청을 중앙에서 상태 check를 거쳐 관리해야 함
- 오버헤드가 큼

## 5. 비교

| 항목 | 한번에 요구 | 순서 할당 | Banker's |
|---|---|---|---|
| 데드락 방지 | O | O | O |
| 병렬성 | 낮음 | 높음 | 중간 |
| 구현 난이도 | 낮음 | 높음 | 높음 |
| 성능 | 낮음 | 높음 | 중간 |
| 현실 적용 가능성 | 낮음 | 높음 | 낮음 |

**(1) 한 번에 요구하기 방식**
- `room_once`로 포크 획득을 원자적으로 처리함
- 사실상 critical section이 커지기 때문에 병렬성이 감소함

**(2) 자원 순서 할당 방식**
- 순환대기를 구조적으로 제거
- 가장 현실적인 해결 방법

**(3) Banker's Algorithm**
- 조건변수 기반 대기
- 상태 기반 제어
- 이론적으로 가장 안전

## 6. 결론 및 느낀점

1의 항목에서 데드락을 직접 경험해보며 데드락의 치명성을 느꼈는데, 이렇게 많은 방식으로 데드락을 방지하는 방법이 있음을 확인하였다.

한 번에 요구하기 방식은 가장 단순하지만 성능이 떨어진다고 느껴졌고, 자원 할당 방식은 데드락을 효과적으로 방지하면서 병렬성을 유지하는 것 같아 가장 효과적이라고 판단했다. 또한 Banker's Algorithm은 이론적으로 당연 가장 안전한 방법이지만 실제 시스템에서 자원이 많아지고 거대해지게 되면 계산 비용과 복잡성 때문에 제한적으로 사용될 수 있겠다는 생각이 들었다.

이번 실습에서 내린 결론은 데드락은 단순한 버그가 아니라 자원 할당 구조의 문제이며, 늘 느끼는 바지만 성능과 안전성은 항상 trade-off 관계에 있구나라고 느꼈다. 실제 시스템이나 현직에선 어떤 방식이 더 자주 쓰이는지 모르지만, 이 역시 상황과 우선순위에 따라 그 중요도가 달라질 것이라 생각한다.
